# 本地 Coding Agent 卡住时，先别怪模型：先判断它到底在等什么

我参与维护开源项目 MA（my-agent）。下面的截图和行为都来自项目当前的实际终端界面，项目仍处于 alpha 阶段。

很多人第一次用本地 Coding Agent，看到界面停在 `thinking...` 十几秒，马上就判断“模型不稳定”。

这通常不够准确。它可能是在加载模型，也可能是在等一个后台工具进程退出，还可能是工具返回太长、生成被截断，或者模型真的开始重复输出。它们表面上都像“卡住”，处理方法却完全不同。

![MA runtime state with model, tools, and context](../website/assets/tui-preview.svg)

我现在用四个状态判断问题在哪：

1. **模型载入**：本地模型第一次调用需要把权重放进内存或显存。这不是推理，也不该被叫成 thinking。
2. **工具执行**：文件读写、命令或浏览器操作需要独立显示工具名、结果和失败原因。工具超时不能伪装成模型在思考。
3. **生成输出**：模型达到输出上限、生成超时或重复同一句话时，系统应收口并保留可恢复的步骤，而不是无限“继续”。
4. **上下文状态**：长任务真正的风险不是“上下文数字变大”，而是日志反复回灌，关键事实被淹没。需要保留结构化证据，而不是把完整输出全塞回 prompt。

![MA init flow and model selection](../website/assets/init-preview.svg)

这也是我做 MA 的原因。它不是另一个把模型包进终端的聊天壳，而是把模型载入、生成、工具调用、上下文和 Agora 本地运行时拆成可观察的阶段。看到状态，才能知道该继续等、该重试工具，还是该停止一次重复生成。

如果你也在做本地 Agent，建议先用一个真实任务压它：读一个仓库、修改文件、启动服务，再验证结果。模型能写几行代码不算完成，任务最后有没有被交付才算。

MA 已 MIT 开源，提供 macOS、Linux 和 Windows 的 alpha 可执行包：<https://github.com/zimoos/my-agent>

我更希望收到真实失败案例，而不是泛泛的“模型不行”。失败案例越具体，Agent 的运行时才能越可靠。
