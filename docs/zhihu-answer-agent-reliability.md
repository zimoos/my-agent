# 知乎问答草稿：Agent 的主战场，不是“能不能调用工具”，而是“能不能把任务做完”

> 适用题目：围绕 Coding Agent、长上下文、工具调用或本地部署的热门问题。
>
> 署名披露：本文作者参与维护开源项目 MA（my-agent），文末会附仓库链接。

我认为 Agent 会成为模型竞争的主战场，但决定体验的不是“模型会不会调用工具”，而是系统能否把一次长任务稳定地收口。

今天大多数模型都可以写代码、读文件、执行命令，也都在宣传 128K、200K 甚至 1M 上下文。真实工作里，用户感受到的失败通常不是“模型不够聪明”，而是下面几件更工程化的事：

1. 工具调用后返回了几千行日志，关键报错被噪声淹没。
2. 一个后台服务没有正确托管，Agent 一直等它退出，表面上像模型卡住。
3. 输出达到上限后继续生成，但继续的是重复文本而不是下一步行动。
4. 本地模型首次加载、显存热身和真正生成混在同一个 `thinking...` 状态里，用户无法判断该等还是该停止。
5. 所谓“记忆”只是把历史文本拼进 prompt，既影响缓存命中，也不能证明模型真的获得了这段状态。

所以我更愿意把 Agent 看成一个运行时系统，而不是一个更长的 prompt。它至少要把五件事做好：

- 给工具结果做证据化和限额，而不是无限原样回灌。
- 将模型载入、生成、工具执行、后台进程和失败恢复作为不同状态展示。
- 对截断、超时和重复输出分别处理，允许任务从可恢复的检查点继续。
- 把长任务是否完成变成可以验证的交付条件，而不是“模型已经说完成了”。
- 对记忆、外部工具和本地运行时返回可验证的状态，而不是只修改系统提示词。

这也是本地模型值得继续做 Agent 的原因。成本、隐私和可控性只是第一层；更关键的是，本地场景逼着我们把运行时的真实边界暴露出来。模型慢、模型要加载、上下文会满、工具会失败，这些在云端演示里容易被掩盖，但在本地必须正面解决。

我参与维护的开源项目 MA 正在做这条路线：支持 LM Studio、DeepSeek 和 Agora 本地运行时；对工具调用、长上下文、后台进程和经 metadata 验证的记忆状态做了明确处理。它还在 alpha 阶段，不把当前基准当成通用排行榜，欢迎用真实失败任务来挑战它。

项目地址：<https://github.com/zimoos/my-agent>
